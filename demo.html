Aurora v4 / CSS core

Ntizar vuelve a empezar para que una app nueva arranque bien desde el minuto uno.

Aurora ya no busca ser un monstruo de clases genericas. Ahora es un CSS pequeno, opt-in y namespaced, pensado para copiar el archivo, activar .nz y construir desde objects, componentes base, glass layers y tokens claros.

.nz root --nz tokens Liquid glass Light + dark
Regla principal del sistema
Primero layout y surfaces. Luego componentes. Aurora v4 no activa su capa visual real hasta que entras por el root .nz.
Ver arranque minimo Ver toda la API
Contrato del core
Lo que define Aurora v4 y lo que ya no promete.
Ruptura limpia
  • Scope: CSS-only, sin build, sin JS de componentes.
  • Aislamiento: toda la API publica usa prefijo nz-.
  • Activacion: los estilos base viven dentro de .nz.
  • Theme: se fuerza con data-nz-theme.
  • Visual language: light-first, azul+naranja y glass como capa expresiva real.
  • Prioridad: apps nuevas, prototypes reales y bloques aislados.
v3 queda atras. Esta demo documenta solo la API nueva.
Learnings

Antes del setup: las reglas de uso que no deberias saltarte

Aurora se entiende mejor desde sus decisiones de uso que desde su lista de clases. Estas son las dos lecciones que mandan sobre el resto de la documentación.

Learning 1 · El design system debe verse
La documentación visual es parte del producto, no un extra.

Si un sistema visual existe pero no aparece al principio de la documentación, para terceros es casi invisible.

  • README para explicar contrato y adopción.
  • Demo para enseñar qué usar, cuándo y cómo.
  • Referencia para cerrar la API pública sin ambigüedades.
Learning 2 · Ntizar.css no entra entero siempre
El core completo es para apps nuevas, no para invadir cualquier proyecto existente.

Si el proyecto ya usa Tailwind, Bootstrap u otro sistema, Aurora no debe entrar entero por defecto.

  • Usa todo el core cuando la app nazca sobre Aurora.
  • Usa tokens o piezas concretas cuando la app ya tenga framework.
  • Usa glass con intención, no como textura automática en todas partes.
Orden correcto para leer esta demo
Primero estos learnings. Luego setup. Después decisión de uso por capa. Y solo al final la referencia completa de clases.
Setup

Arranque minimo en 3 pasos

Si el proyecto nace sobre Aurora, no necesitas mas. Copias el archivo, activas el root y decides si mantienes el tema claro o si fuerzas dark.

1. Copia `ntizar.css`
Cuando: siempre que el proyecto nuevo quiera usar Aurora como base.
No lo hagas asi: no mezcles la API nueva con clases legacy tipo .card o .btn.
2. Activa `.nz`
Cuando: en todo el body si la app nace con Aurora, o en un wrapper si quieres aislar una zona.
No lo hagas asi: no esperes que Aurora estilice nada fuera de .nz.
3. Decide tema y tokens
Cuando: usa light por defecto; fuerza dark solo si el producto lo necesita.
No lo hagas asi: no edites el core para recolorear la marca; sobreescribe tokens.
HTML Base minima para una app nueva 
<!DOCTYPE html>
<html lang="es">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">

</html>
Limite deliberado de v4
Aurora v4 incluye menos piezas que v3. Es intencional. El core prioriza contrato limpio y adopcion simple antes que una coleccion enorme de patrones.
Que usar

Las decisiones que tomas una y otra vez

La mayoria del valor de Aurora esta en elegir bien la capa. Este bloque responde al "cuando" y al "como" antes de entrar al detalle de cada clase.

Opcion Usala para Evitala cuando
.nz-surface Paneles neutros, filtros, grupos de informacion o sidebars. Necesites contenido con header/body/footer reutilizable; ahi conviene .nz-card.
.nz-surface--glass Bloques destacados, heroes, demos, zonas premium o una capa visual con mas presencia. La pantalla ya tenga demasiada textura o si todo se vuelve visualmente pesado.
.nz-surface--glass-brand Paneles con ADN Ntizar mas marcado: dashboards principales, heroes o estados activos. Necesites una superficie neutra o documental.
.nz-card Items de dashboard, resultados, summaries, modulos reutilizables. Solo quieras agrupar contenido sin semantica de card; usa surface.
.nz-card--interactive Cards clicables, listados o piezas que deben sentirse activas. La card no sea interactiva; el hover extra anade ruido innecesario.
.nz-btn--primary La accion principal del bloque o formulario. Tengas varias acciones con el mismo peso; entonces usa secondary o ghost.
.nz-btn--ghost Cancelar, atras, cerrar o acciones auxiliares. La accion deba competir visualmente con el CTA principal.
.nz-btn--glass* Toolbars, acciones premium o contextos con glass sin perder legibilidad. El CTA principal deba leerse inmediato a distancia; ahi gana primary o accent.
.nz-alert--warning Riesgo moderado, aviso operacional o decision que requiere atencion. No haya una consecuencia clara; en ese caso mejor callout o ayuda inline.
.nz-callout--tip Explicaciones, notas de docs, onboarding o guidance de IA. Necesites feedback de estado del sistema; para eso usa alert.
Objects

Layout antes que componentes

Aurora v4 arranca por estructura. Si el layout esta bien resuelto, los componentes se mantienen pocos y claros.

Container + Section
El borde exterior de cada pantalla.
Empieza casi siempre con .nz-section y dentro mete un .nz-container. Eso ya te da ritmo vertical y anchura consistente.
Stack
La solucion por defecto para apilar contenido.
Usa .nz-stack para bloques verticales. Cambia el ritmo con .nz-stack--sm o .nz-stack--lg antes de meter margins manuales.
Grid
Para paneles, dashboards y listas visuales.
.nz-grid--2 y .nz-grid--3 cubren la mayoria de pantallas. .nz-grid--aside sirve para docs, settings o un dashboard con sidebar.

Liquid glass real en el core

Aurora no vuelve al exceso de v3, pero tampoco renuncia a su lenguaje visual. El glass ahora es una familia concreta de surfaces y cards, con niveles y tintes claros.

Soft
Glass soft
Para overlays tranquilos, filtros o bloques que solo necesitan profundidad ligera.
Standard
Glass
La variante general para cards destacadas, promos y paneles premium.
Strong
Glass strong
Para hero cards o bloques donde el glass debe sentirse mas denso y cinematografico.
Brand
Glass brand
Azul, violeta y naranja integrados como parte del look Aurora.
Accent
Glass accent
Para CTAs secundarios, upsells, banners y elementos mas calidos.
Components

El set minimo que si merece vivir en el core

Botones, cards, fields y feedback cubren casi cualquier primer sprint de producto. Lo importante es que cada pieza tenga una responsabilidad clara.

Botones

La escala sigue siendo corta, pero ahora vuelve el lenguaje Aurora: primario azul, mezcla de marca y variantes glass de verdad.

Regla de uso
Dentro de un mismo bloque intenta no combinar mas de un primary y un ghost. Si todo compite, nada destaca.

Fields

El formulario base cubre inputs, selects, textareas y ayuda inline.

Usa help para guidance normal.
Marca error solo cuando haya un problema concreto que resolver.
Badges
Meta, estado corto y densidad baja.
Draft Core Launch Glass Aurora Warm Stable Review Blocked

Badge para una palabra o dos. Si el mensaje necesita contexto, pasa a alert o callout.

Feedback
Estado del sistema frente a guidance de contenido.
Info
Usa alert cuando el sistema este informando de un estado o resultado.
Warning
Si el riesgo es real pero aun reversible, warning suele ser la mejor variante.
Tip
Usa callout para explicar criterios de uso, onboarding o contexto documental.
Theming

Light por defecto. Dark solo cuando el producto lo pide.

El tema ya no depende de clases desperdigadas por componentes. Aurora usa una decision explicita en el root y un grupo pequeno de tokens para personalizacion de marca.

Reglas de tema

  • Default: light. Es la base recomendada para producto nuevo.
  • Dark: fuerzalo con data-nz-theme="dark".
  • Marca: cambia --nz-color-brand, --nz-color-accent y, si hace falta, --nz-gradient-aurora.
  • No hagas: no dupliques componentes para cada tema.
CSS Sobrescribe tokens despues del core 
.mi-shell.nz {
  --nz-color-brand: #6d28d9;
  --nz-color-brand-strong: #5b21b6;
  --nz-color-accent: #ec4899;
  --nz-color-accent-strong: #db2777;
  --nz-gradient-aurora: linear-gradient(135deg, #2563eb 0%, #7c3aed 48%, #ec4899 100%);
  --nz-radius-lg: 1rem;
  --nz-font-sans: "Poppins", system-ui, sans-serif;
}
